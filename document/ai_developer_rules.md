# CityWork 影像资产管理平台 - AI 开发者规则与行为准则

作为协助开发 CityWork 影像资产管理平台的 AI，你必须**严格遵守**以下规则。这些规则旨在确保开发过程的严谨性、一致性以及与现有架构设计的完全对齐。

## 1. 强制性前置阅读规则

- **规则 1.1：全局上下文认知**：在**编写任何代码或执行任何构建命令之前**，你必须首先完整阅读 `d:\Front-end\Vue\code\ai-picture\document` 文件夹下的所有核心文档，包括：
  - `frontend_architecture_design.md` (架构设计文档)
  - `tech_stack_selection.md` (技术栈选型文档)
  - `implementation_plan.md` (实施计划文档)
- **规则 1.2：遵循既定设计**：禁止引入上述文档中未提及的核心技术栈、框架或设计模式。如果遇到文档未覆盖的边界情况，必须先向用户确认，或基于现有架构风格（简单、健壮、Vue 3 组合式 API）进行合理推演，并说明理由。

## 2. 实施计划与进度同步规则

- **规则 2.1：严格按步执行**：开发工作必须严格按照 `implementation_plan.md` 中的阶段和步骤顺序进行。除非用户明确指示，否则严禁跳过步骤或打乱顺序。
- **规则 2.2：强制测试验证**：在完成 `implementation_plan.md` 中的任何一个子步骤后，必须执行该步骤对应的“**验证**”指令。只有验证通过，才算真正完成该步骤。
- **规则 2.3：同步更新进度文档**：每完成并验证通过一个小步骤（例如“步骤 1.1”），**必须立即**更新 `d:\Front-end\Vue\code\ai-picture\document\progress.md` 文件。
  - **记录格式要求**：在 `progress.md` 中记录完成的时间、对应的步骤编号及名称、以及验证结果的简要说明。

## 3. 架构与文件约束规则

- **规则 3.1：同步更新架构文档**：在开发过程中，每当你创建了新的核心目录、重要组件文件（如视图组件、Store、工具类等）或全局配置文件时，**必须同步**更新 `d:\Front-end\Vue\code\ai-picture\document\architecture.md` 文件。
  - **记录格式要求**：在 `architecture.md` 中以树状结构或清晰的列表形式，记录新增文件的绝对路径及其具体的业务作用与功能边界。
- **规则 3.2：按 Domain 划分 Store**：严格遵守状态管理的 Domain 划分。只能创建 `UserStore`、`SpaceStore` 和 `PictureStore`，禁止随意创建碎片化的状态文件。
- **规则 3.3：Mock 数据隔离**：所有因缺少后端接口而使用的 Mock 数据（如城市列表、团队故事），必须统一放置在根目录的 `constant` 文件夹中，严禁在组件内部硬编码大段静态数据。

## 4. 交互与反馈规则

- **规则 4.1：小步快跑与确认**：在执行较为复杂或涉及多个文件修改的步骤时，完成该步骤后应向用户简要汇报修改了哪些文件及验证结果，并询问是否继续进行下一步。
- **规则 4.2：非破坏性修改**：对于现有文件的修改，优先使用精准的正则替换（SearchReplace）或局部更新，避免在没有充分理由的情况下重写整个文件。
