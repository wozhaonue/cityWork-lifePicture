# CityWork 影像资产管理平台 - AI 开发者前端实施计划

## 概述

本文档是基于《前端架构设计文档》和《前端技术栈选型文档》制定的严格实施计划。该计划专门为 AI 开发者设计，所有的步骤被拆分为小而具体的指令，不包含任何代码，且每一步都附带了明确的验证标准。AI 开发者在执行时需严格按照顺序逐步推进。

> **🎨 UI 视觉参考全局指令**：
> 在执行任何涉及 UI 页面重构或组件样式开发的步骤时，若对“数字胶片”与“现代极简”的样式风格把握不够准确，**必须**查看并参考 `d:\Front-end\Vue\code\ai-picture\instance\screenshots` 目录下的页面截图作为最终的 UI 视觉验证基准。

## 阶段一：项目初始化与基础设施建设

### 步骤 1.1：初始化核心框架

- **指令**：使用 Vite 初始化 Vue 3 和 TypeScript 模板的前端项目，配置包管理器（如 npm）。
- **验证**：运行本地开发服务器，确保浏览器能够正常渲染 Vue 的默认欢迎页面，且浏览器控制台无任何报错信息。

### 步骤 1.2：安装并配置 UI 组件库

- **指令**：安装 Element Plus 及其相关的按需引入插件，并配置在项目中全局可用。
- **验证**：在根组件中放置一个 Element Plus 的主按钮组件，刷新页面检查该按钮是否渲染成功且带有正确的默认样式。

### 步骤 1.3：安装并配置样式方案

- **指令**：安装 Tailwind CSS 及其相关依赖（PostCSS、Autoprefixer），初始化配置文件。在 `tailwind.config.js` 中配置自定义的“数字胶片”与“现代极简”设计规范：配置背景主色调（如磨砂白 `#fcfbf9`）、大地色系/高级灰辅助色，并在全局样式文件中引入 `Noto Serif SC` 衬线字体以提升排版质感。配置 SCSS 支持。
- **验证**：给刚才添加的按钮加上 Tailwind 的背景色和文字颜色类名，检查浏览器中该按钮是否成功应用了对应的颜色变化，且页面整体背景和字体已应用新配置的样式。

### 步骤 1.4：配置插件化网络请求层

- **指令**：安装 Axios。采用**插件化架构**设计 HTTP 客户端实例。创建核心 Axios 实例，并编写独立的拦截器插件：
  1. **Auth Plugin**：配置跨域请求携带 Cookie（`withCredentials: true`）及处理未登录状态码（如 401 跳转登录页）与路由守卫联动。
  2. **Error Plugin**：统一处理错误状态码并结合 Element Plus 弹窗提示。
  3. **Retry & Deduplication Plugin**：实现请求去重（防止快速连点）与超时重试机制。
- **验证**：在入口组件中连续发起三次相同的模拟数据请求，在浏览器的 Network 面板中检查是否只有一次实际的网络请求发出（去重插件生效）；发起一个会超时的请求，验证是否会自动重试并最终弹出错误提示框。

## 阶段二：状态管理与全局静态资源

### 步骤 2.1：初始化路由和状态管理

- **指令**：安装 Vue Router 4 和 Pinia。创建基础的路由实例并挂载到 Vue 应用上。创建 Pinia 实例并挂载。根据架构设计创建 UserStore、SpaceStore 和 PictureStore 的空壳文件。
- **验证**：打开浏览器的 Vue DevTools 扩展，检查 Router 和 Pinia 面板是否已被成功识别且状态为空。

### 步骤 2.2：构建全局城市静态数据

- **指令**：在项目根目录下创建常量文件夹（constant）。在其中创建存储中国所有地级市名称的静态数组文件并导出该数组。
- **验证**：在首页组件中引入该常量数组，将数组的长度打印在页面上，检查显示的数字是否符合预期。

### 步骤 2.3：构建全局团队故事 Mock 数据

- **指令**：在常量文件夹中创建用于模拟团队故事数据的静态对象数组，包含团队 ID、团队名称、故事摘要和故事详情页的模拟链接。
- **验证**：在控制台中打印该数据对象，检查数据结构是否符合预期。

### 步骤 2.4：实现全局图片懒加载自定义指令

- **指令**：在项目中创建 `directives` 目录，编写全局自定义指令 `v-image-pro`。使用 `IntersectionObserver` 监听图片进入可视区后才赋予 `src` 属性进行加载；添加 `decoding="async"` 提升渲染性能；处理 `onerror` 事件，实现图片加载失败时的自动重试逻辑及最终的默认兜底图替换。
- **验证**：在任意页面渲染包含 50 张图片的列表并绑定 `v-image-pro`，在浏览器 Network 面板验证：只有滚动到可视区的图片才发起网络请求；人为修改一张图片的 src 为错误地址，验证是否能自动重试并最终显示兜底图。

## 阶段三：身份认证、核心布局与路由守卫

### 步骤 3.1：实现登录与注册页面

- **指令**：在项目中创建用户登录和注册页面，使用 Element Plus 的表单组件实现账号、密码等字段的输入与校验。
- **验证**：在页面中尝试输入不符合规则的密码或留空必填项，验证表单校验是否正常拦截并提示。

### 步骤 3.2：集成状态保持与请求鉴权

- **指令**：编写对接 `frontend/src/api/userController.ts` 的登录 API 的逻辑。基于 Session-Cookie 机制，登录成功后后端自动通过 `Set-Cookie` 下发凭证。当登录成功后，即 `res.data.code === 0` 时，调用 `frontend/src/store/modules/user.ts` 中的 `setUserInfo` 方法来进行用户信息的记录。前端需确保 Axios 实例已配置跨域携带凭证（`withCredentials: true`）。在业务代码的 `catch` 异常捕获块中，需过滤掉因防抖插件触发的取消异常（`axios.isCancel`）。
- **验证**：
  1. 使用真实测试账号（账号：`wsb01`，密码：`Wsbxy132.`）执行一次登录请求，验证页面是否提示登录成功、跳转至首页，并在浏览器的 Network 面板中检查请求是否成功。
  2. 输入错误的账号密码，**快速连续点击登录按钮**，验证是否只会触发一次网络请求，并且页面上只弹出一次后端返回的“用户不存在或者密码错误”的业务提示，而不会弹出前端 `catch` 块中的兜底“登录异常”提示。

### 步骤 3.3：搭建页面基础路由

- **指令**：配置核心顶级路由页面：登录页、注册页、个人空间页面、团队空间页面、公共图库页面、统一管理中枢页面。
- **验证**：在浏览器地址栏手动输入上述页面的 URL，验证是否能够成功匹配并展示对应的占位文本，而不触发 404 错误。

### 步骤 3.4：实现全局布局结构

- **指令**：创建一个基础布局组件。布局需遵循“杂志排版 (Editorial)”风格：提供大面积留白，包含顶部隐藏式或极简的导航栏（仅展示必要的 Logo、纯文本路由链接和用户入口），主内容区域不设生硬的边框。将空间和图库等业务路由配置为该布局的子路由。
- **验证**：登录后点击顶部导航栏的不同链接，验证主内容区域是否能够正确切换页面，整体页面视觉呈现出呼吸感。

### 步骤 3.5：实现路由鉴权守卫

- **指令**：配置全局路由前置守卫（`beforeEach`）。对于需要登录的页面（如个人/团队空间、管理中枢），检查 UserStore 中是否存在有效登录状态，若无则强制重定向至登录页。同时，针对管理中枢页面，检查用户角色是否为管理员，若不是则重定向至公共图库。
- **验证**：在未登录状态下尝试访问个人空间路由，验证是否被强制跳转到登录页。在普通用户状态下尝试访问管理中枢路由，验证是否被跳转到公共图库。

## 阶段四：个人空间与基础流转

### 步骤 4.1：实现强制城市标签的图片上传表单

- **指令**：在个人空间中实现基于 Element Plus 的图片上传表单。表单必须包含一个支持搜索的下拉选择框，其数据源绑定为阶段二创建的城市静态数组。将其设置为必填项。
- **验证**：尝试在不选择城市的情况下提交表单，验证表单校验是否拦截提交并显示错误提示。选择一个城市后提交，验证是否能够顺利通过校验。

### 步骤 4.2：实现图片文件夹归类视图

- **指令**：在个人空间中实现具有“拍立得 (Polaroid)”照片墙风格的图片分组视图。图片不采用整齐的网格，而是通过 Tailwind 配置不同的大小和轻微的角度旋转（`rotate`）形成非对称散落感。支持鼠标 Hover 时的微交互：从灰度滤镜平滑过渡到彩色，模拟显影过程。每一组使用衬线体城市名称作为标题。
- **验证**：在状态中注入几条带有不同城市标签的模拟图片数据。检查页面是否按照城市标签分组，且图片呈现出参差不齐的“相片”质感，Hover 时具有自然的显影微交互。

### 步骤 4.3：实现资产跨空间流转交互

- **指令**：为每一张图片实现一个“流转副本”的按钮交互。点击后弹出一个对话框，允许用户选择目标空间（公共图库或特定的团队空间）。
- **验证**：点击任意图片的流转按钮，检查对话框是否正常弹出。选择目标空间并确认后，验证是否触发了预期的流转动作（可通过控制台日志或页面通知确认）。

## 阶段五：团队空间与协作基础

### 步骤 5.1：实现团队成员管理 UI

- **指令**：在团队空间页面中实现一个成员列表表格，展示成员信息和角色。为管理员角色添加可以切换其他成员角色（管理员/普通成员）和移除成员的操作按钮。
- **验证**：使用模拟数据渲染列表。点击切换角色按钮，验证界面上的角色标签是否即时更新，且控制台输出对应的模拟接口调用日志。

### 步骤 5.2：搭建异步协作画布占位

- **指令**：在团队空间中创建一个图片编辑页面组件（WorkspaceCanvas）。应用“暗房 (Darkroom)”概念：配置深色的径向渐变背景，叠加一层带有毛玻璃效果（`backdrop-filter: blur`）的浮动操作面板，面板中放置编辑与模拟 AI 扩图按钮。
- **验证**：进入该组件页面，验证整体背景是否呈深色暗房效果，且面板带有毛玻璃质感。点击模拟操作按钮，验证是否显示平滑优雅的 Loading 状态并最终提示成功。

## 阶段六：公共图库与 3D 地图交互

### 步骤 6.1：集成 ECharts-GL 构建 3D 地图底座

- **指令**：安装 ECharts 及 ECharts-GL（必须使用 GL 才能支持真 3D 渲染，不可使用纯 CSS 伪造）。在公共图库页面中，初始化全屏背景画布，加载中国地图的 `china.js` GeoJSON 数据。配置 `geo3D`，设置材质颜色为“高级灰/大地色”，设置视角支持鼠标自适应旋转 (`autoRotate`)，背景必须为透明（`transparent`）或极简磨砂色。
- **验证**：进入公共图库页面，验证页面背景是否呈现出了具备真 3D 景深与阴影效果的中国地图模型，且用户可以通过鼠标任意旋转拖拽该地图。

### 步骤 6.2：实现视差滚动 (Scroll-telling) 影像展示

- **指令**：基于 3D 地图底图，在顶层实现一个透明的、可滚动的“影像画报”层（`z-index` 高于地图）。随着用户的滚轮滑动，动态加载城市照片并展示。前端利用 `GSAP` 或纯 CSS `transform: translateY` 实现前后景移动速度不一致的视差滚动效果。
- **验证**：向下滚动页面，验证图片是否像杂志般平滑浮现，同时背景 3D 地图保持固定或以更缓慢的速度旋转/位移，产生强烈的“时空漫步”沉浸感。

### 步骤 6.3：实现叙事化标注与动态配色

- **指令**：获取 3D 地图上存在影像资产的城市坐标，利用 ECharts-GL 的 `scatter3D` 渲染出黑白质感的光点或极简图钉。当点击某张图片或光点时，使用“叙事化标注”（如：浮现一段右对齐的心情笔记文本弹幕）。可选实现基于当前预览图片主色的“智能环境配色”，即调整周边的阴影或按钮高光。
- **验证**：点击城市光点，验证出现的不再是传统 POI 信息框，而是带有情感语气的悬浮文本和相关图片。

## 阶段七：管理中枢与数据分析

### 步骤 7.1：实现极简无边框数据大盘

- **指令**：在管理中枢的数据分析页面中，利用 ECharts（2D）实现空间容量分布饼图和热门城市柱状图。整个管理中枢应摒弃 Element Plus 的原生强边框样式，通过覆盖样式（或采用 Tailwind）实现全白背景、无边框表格、大间距以及带下划虚线（`border-dashed`）的杂志风列表。
- **验证**：验证大盘页面是否显得极为“干净”，图表和列表仿佛直接印刷在白纸上，而没有被各种灰色线框框死。

### 步骤 7.2：实现图片审核工作流

- **指令**：在管理中枢的图片审核页面中实现一个待审核图片列表。每一项包含图片预览、来源信息以及“通过”和“驳回”按钮。
- **验证**：点击列表中某张图片的“通过”按钮，验证该图片是否从待审核列表中立即消失，并弹出审核成功的提示信息。

## 阶段八：性能优化与发布准备

### 步骤 8.1：实施路由与组件懒加载

- **指令**：将 Vue Router 的路由配置全部修改为动态导入（懒加载）模式。确保公共图库的 3D 引擎相关代码被打包为独立的 Chunk。
- **验证**：在浏览器的 Network 面板中，清空缓存并刷新页面。验证切换不同顶级路由时，才会发起新的 JS 文件请求，而不是在首页一次性加载所有代码。

### 步骤 8.2：实施 WebGL 内存防泄漏机制

- **指令**：在公共图库的 3D 地图组件的卸载生命周期钩子中，编写销毁 ECharts 实例及清理相关 WebGL 上下文的逻辑。
- **验证**：在公共图库页面和其他普通页面之间反复切换 10 次以上。通过浏览器的性能分析工具（Performance/Memory）检查内存堆快照，验证内存占用是否保持稳定，没有出现持续增长的泄漏现象。
