# CityWork 影像资产管理平台 - 前端架构设计文档

## 1. 概述 (Overview)

**项目背景**：针对 CityWork 场景的影像资产管理平台，旨在为摄影师、城市记录者及团队提供一个从个人作品沉淀、团队协作共创，到公共图库3D大屏展示的全链路闭环系统。
**核心目标**：

- **资产隔离与流转**：实现个人空间、团队空间、公共图库的权限隔离，支持图片以“复制副本”形式跨空间流转。
- **3D 可视化展示**：公共图库采用 3D 中国地图，提供极具视觉冲击力的城市影像导航体验。
- **平滑迭代演进**：架构初期基于现有 API 进行降级设计（常规直传、异步协同），并在代码架构上为未来“大文件分片上传”与“实时画布协同”预留无缝升级的扩展位。

---

## 2. 技术栈选型 (Technology Stack)

为了保证项目的高维护性与前沿体验，推荐以下前端技术矩阵：

- **核心框架**：`Vue 3` (Composition API, `<script setup>`) + `TypeScript` (强类型约束，提升复杂业务稳定性)。
- **构建工具**：`Vite` (极速冷启动与热更新，针对 3D 资源加载可配置相关 plugin)。
- **路由与状态管理**：`Vue Router 4` (支持动态路由注入) + `Pinia` (轻量化状态管理，按 Domain 划分 Store)。
- **UI 组件库**：`Element Plus` (成熟的后台管理与表单组件，契合复杂的数据录入需求)。
- **3D 与可视化引擎**：`Three.js` 搭配 `ECharts 3D`。用于构建公共图库的 3D 立体中国地图，支持城市节点的 3D 气泡/光柱交互。
- **网络层**：`Axios`，采用**插件化架构**（Plugin-based Architecture），将请求去重、权限校验、重试机制等抽象为独立插件，提升网络层扩展性与可维护性。
- **样式方案**：`Tailwind CSS` (原子化 CSS，加速 UI 构建) + `SCSS` (处理 3D 容器及复杂动画的局部样式)。

## 3. UI/UX 设计规范 (UI/UX Design Specification)

系统整体采用 **“数字胶片 (Digital Film)”** 与 **“现代极简 + 杂志排版 (Modern Editorial)”** 的核心视觉风格，旨在为用户提供具有呼吸感和沉浸感的影像浏览体验，摒弃传统后台的生硬感。

> **🎨 视觉参考规范 (Visual Reference Rule)**：
> 在后续开发中，若对于页面整体布局、UI 样式细节、排版比例或微交互的把握不够准确，**必须优先参考** `d:\Front-end\Vue\code\ai-picture\instance\screenshots` 目录下的核心页面截图，以该文件夹中的页面展示状态作为 UI 样式的绝对基准。

### 3.1 核心视觉语言

- **色彩体系**：拒绝荧光色，以“高级灰”和“大地色系”为底色，背景主色调推荐采用柔和的磨砂白（如 `#fcfbf9`）。配合智能环境配色（Dynamic Color Extraction），UI 辅助色需根据当前影像自动提取，确保 UI 与影像内容的完美融合。
- **排版逻辑**：强化大留白与非对称布局。影像资产不采用生硬的等宽切片，而是类似拍立得照片或艺术杂志般错落有致，字体推荐使用衬线体（如 `Noto Serif SC`）提升文艺质感。
- **微交互 (Micro-interactions)**：所有的交互需轻盈自然。例如，点击图片时产生轻微浮起效果（如 `rotate` 和 `scale` 组合变换），背景适度变暗，避免生硬的弹窗。使用毛玻璃效果（Glassmorphism，`backdrop-filter`）增强层级感。

### 3.2 页面级设计范式

- **公共图库 (`/public`)**：
  - **3D 沉浸式地图**：采用深色或低饱和度的 3D 地球/地图底座（ECharts-GL），配合悬浮的光点展示城市影像分布。
  - **视差滚动 (Scroll-telling)**：抛弃传统的点击跳转翻页，采用无缝滚动加载。前景内容与背景影像以不同速度移动，形成空间深度感。
  - **叙事化标注 (Narrative Markers)**：采用弹幕或心情笔记形式的悬浮气泡，替代传统生硬的 POI 图钉。
  - **声音滤镜 (Spatial Audio)**（可选增强）：在浏览特定城市影像时，后台播放该城市的空间环境音（如白噪音）。
- **个人空间 (`/personal`)**：采用“拍立得”照片墙的非对称散落布局，支持鼠标 Hover 时从灰度图平滑过渡到彩色，模拟照片显影过程。
- **团队空间 (`/team`)**：采用“暗房 (Darkroom)”概念，深色渐变背景叠加毛玻璃质感的操作面板，营造专注的协同编辑氛围。
- **管理中枢 (`/admin`)**：采用无边框表格、大间距和全大写英文字母作为表头（杂志风排版），用极简的线条（如下划虚线）划分数据边界。

## 4. 模块化设计与功能边界 (Modular Design)

### 4.1 空间引擎模块 (Space Engine)

- **个人空间 (`/personal`)**
  - **职责**：个人高清原片的私密保险箱。
  - **核心功能**：
    - 图片上传与编辑：图片上传底层封装 `Uploader` 类。在添加和编辑图片信息时，表单内强制要求填写所属城市（采用带搜索功能的选择框，数据源为前端 `constant` 文件夹下存储的包含中国所有地级市的静态数组）。选中的城市信息将作为标签（`tags`）附加到图片实体上进行流转。
    - 城市文件夹：基于前述的城市标签对空间内的照片进行归类展示。
    - 资产流转（选中图片 -> 复制副本 -> 更改新副本的 `spaceId` 并流转至团队/公共空间）。
- **团队空间 (`/team/:spaceId`)**
  - **职责**：团队成员资源共享与协同作业。
  - **核心功能**：
    - 成员管理：基于 `space-user-controller` 实现成员邀请与 `spaceRole` (管理员/普通成员) 的变更。
    - 协同编辑：当前基于权限判定（`permissionList`），允许多人异步调用 `editPicture` 或 AI 扩图任务。前端架构将画布抽象为 `WorkspaceCanvas` 组件，以便未来接入 WebSocket 和 CRDT 算法升级为实时协同。
- **公共图库 (`/public`)**
  - **职责**：经过审核（`reviewStatus = 1`）的城市级图片大屏展示。
  - **核心功能**：
    - 3D 地图动态点位展示：前端获取图库所有图片的 `tags` 后，与 `constant` 中的城市静态数组进行交集过滤，仅在 3D 地图上渲染存在对应图片资产的城市点位（无图片的城市保持隐去状态）。鼠标 Hover/Click 城市节点弹出该城市的精选缩略图，点击“进入图库”跳转至城市相册瀑布流（`/public/city/:cityName`）。
    - 团队故事溯源：在公共图库的照片瀑布流/详情展示中，增加发布团队的署名信息，并在图片下方提供悬浮的“团队故事”气泡。点击气泡可进入该团队的专栏页面。（注：因初期无后端接口支持，团队故事的展示页采用纯静态页面实现，数据由前端进行 Mock）。

### 4.2 管理中枢模块 (Admin Center)

- **统一入口**：全局布局组件中判定 `user.userRole === 'admin'`，渲染悬浮或顶部专属入口按钮。
- **子页面 (路由鉴权拦截)**：
  - **数据分析 (`/admin/dashboard`)**：对接 `space-analyze-controller`，利用 ECharts 绘制各空间容量、标签分布等分析图表。
  - **图片审核 (`/admin/pictures`)**：重点拦截流转到公共图库（`spaceId = 0/null` 且 `reviewStatus = 0`）的图片，调用 `doPictureReview` 进行审批。
  - **用户与空间管理 (`/admin/users`, `/admin/spaces`)**：宏观掌控系统内账号状态及异常空间的管控。

### 4.3 身份认证与用户中心模块 (Identity & User Center)

- **职责**：负责全局用户的账号注册、登录鉴权、状态保持及个人信息维护，是整个系统的前置守卫。
- **核心功能**：
  - **登录与注册 (`/user/login`, `/user/register`)**：对接 `user-controller` 实现鉴权。基于 Session-Cookie 机制，登录成功后后端通过响应头 `Set-Cookie` 下发 Session ID，浏览器自动保存并在后续请求头中自动携带，前端无需手动维护 Token。前端仅需在状态库中保存当前登录用户信息（`UserVO`）。
  - **状态初始化与路由守卫**：在应用初始化或路由跳转时，获取当前登录用户信息（`getLoginUser`），写入 `UserStore`。结合 Vue Router 全局前置守卫（`beforeEach`），拦截未登录用户对空间或管理中枢的访问，强制重定向至登录页。
  - **个人中心 (`/user/profile`)**：提供用户基础信息（如头像、昵称、简介等）的展示与修改闭环。

## 5. 核心数据模型与 Store 设计 (Data Model & Store)

前端状态抽象为三大 Domain Store：

1. **UserStore**:
   - `currentUser`: 存储当前登录用户的 `UserVO`（含 `userRole` 用于路由守卫）。
2. **SpaceStore**:
   - `currentSpace`: 当前正在浏览的 `SpaceVO` (个人/团队)。
   - `teamMembers`: 当前团队的 `SpaceUserVO[]` 列表。
3. **PictureStore**:
   - 定义标准的前端实体 `FrontendPicture`，适配后端的 `PictureVO`。
   - 包含鉴权 Getter：`canEdit(picture)`，结合图片自身的 `userId` 与当前空间的 `spaceRole` 计算是否渲染编辑按钮。

## 6. 关键链路交互流 (Interaction Flow)

### 6.1 影像流转与审核链路

1. **发起流转**：用户在个人空间选中 ID 为 `101` 的照片，点击“发布到公共图库”。
2. **生成副本**：前端调用后端（当前需通过下载再上传，或推动后端新增一键 Copy 接口）生成一张全新图片记录 ID `102`。
3. **状态标记**：ID `102` 的图片 `spaceId` 被置空（归属公共），且 `reviewStatus = 0`（待审核）。
4. **后台审核**：管理员进入统一入口 -> 图片审核 -> 预览 `102` 无违规 -> 调用 `/api/picture/review` 变更为 `1`（通过）。
5. **地图展示**：普通用户进入首页 3D 地图，点击对应城市触发 `listPictureVOByPage` (入参 `category=城市名`, `reviewStatus=1`, `spaceId=null`) 并在弹窗展示。

### 6.2 3D 地图与数据下钻交互

1. **加载引擎**：加载 Three.js 场景、相机、光照及中国 GeoJSON 数据，生成 3D Mesh。
2. **数据聚合与点位过滤**：前端提取所有图片中包含的城市 `tags`，与本地城市静态数组进行交集匹配，得出有效城市列表。接着在 3D Mesh 对应经纬度仅生成有效城市的 3D 柱状图/光点（高度映射数量），无数据的城市则隐去点位。
3. **Raycaster 拾取**：监听 Canvas 的 Click 事件，利用 Raycaster 射线相交算法计算用户点击的城市。
4. **2D UI 叠加**：计算 3D 坐标到屏幕 2D 坐标的投影，利用 Vue 的 `<Teleport>` 或绝对定位弹出包含 `thumbnailUrl` 的浮层组件。

## 7. 部署与扩展性 (Deployment & Scalability)

### 7.1 工程化与 CI/CD

- **多环境配置**：通过 `.env.development`, `.env.production` 隔离本地调试（指向 `localhost:8123`）与线上网关。
- **容器化部署**：编写 `Dockerfile`，第一阶段 Node.js 镜像执行 `pnpm build`，第二阶段使用 Nginx Alpine 镜像托管 `dist` 目录并配置 `try_files $uri /index.html` 以支持 History 路由。

### 7.2 性能优化基准 (Performance)

- **3D 资源按需加载**：Three.js 核心库及地图模型较重，将 `/public` 路由设定为独立 Chunk，并在进入前展示炫酷的 Loading 动画。模型纹理采用 CDN 加速。
- **极致图片加载机制 (自定义指令)**：封装全局图片渲染指令 `v-image-pro`。结合 `IntersectionObserver` 实现智能懒加载；利用 `decoding="async"` 属性将解码移出主线程防止卡顿；内置加载失败的重试逻辑与默认兜底图，并支持平滑过渡动画。
- **图片渐进式渲染**：列表页、地图弹窗**严格**使用后端的 `thumbnailUrl` 字段渲染；仅在进入详情页、协同编辑页时加载 `url` 高清原图。
- **防内存泄漏**：在组件 `onBeforeUnmount` 阶段，必须手动调用 Three.js 的 `geometry.dispose()` 与 `material.dispose()` 清理 WebGL 内存。
